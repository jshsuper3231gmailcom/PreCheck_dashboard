# PreCheck 로그 수집 스케쥴정의서 v1.0

## 문서의 목적
- 로그 수집 시스템에서 참고할 수집 대상서버, 대상서버내 파일 위치/수집 주기를 정의한 파일을 설명
- 로그 수집 시스템내 파일 형태로 정의함

---

## 로그 수집 정의서 파일명
- 로그 수집 시스템 내 ~/cfg/PreCheck_CollectLogs_Schedule.conf

## 로그 수집 주기
- 로그 수집 주기는 다음과 같음
- 배치 -> 일회성 로그 수집, 로그 전체
- 주기 -> 정해진 시간부터 주기별로 로그 수집, 최초 수집은 로그 파일 전체, 이후 주기부터는 추가된 로그만을 수집(마지막으로 읽은 라인번호로 판별)

## 로그 수집 정의 파일 포맷
- 로그 수집 정의 파일의 포맷은 다음과 같음
- [서버구분][아이피주소][대상파일명(절대경로)][수집주기기술]
    -> 서버구분 : 대상서버의 호스트명 또는 서버사용명(ex 서버자동주문)
    -> 아이피주소 : 대상서버의 아이피주소
    -> 대상파일명 : 대상서버내 로그 위치, 절대경로
    -> 수집주기기술 : 수집구분, 수집시간, 수집주기

- 수집주기기술은 수집구분별로 다음의 포맷으로 정의
    -> 배치 : [배치|요일|시작시간]
    -> 주기 : [주기|요일|시작시간|수집간격(분)|종료시간]
        -> 요일 표기 규칙
            * 단일 요일 : 0~6 중 하나 (0->일요일, 1->월요일, 2->화요일, ..., 6->토요일)
            * 범위      : 시작-끝 형식 (예: 1-5 = 월~금)
                          - 시작은 끝보다 작거나 같아야 함
                          - 0-6은 *과 동일 (매일)
            * 전체      : * (매일)
            * 콤마(,) 또는 복합 표기(예: 1,3,5 또는 1-3,5)는 1차 버전에서 미지원
            * 위 규칙을 따르지 않는 요일 표기는 스케쥴 무시 처리
        -> 시작시간 : HHmmss, 시작시간은 000001이 최소임
        -> 수집간격(분) : 분단위(예 5:5분)
        -> 종료시간 : HHmmss, 종료시간은 235959가 최대임

- 파일은 텍스트 형태며 개행문자(\n)로 구분
- 위 포맷을 따르지 않는 스케쥴은 무시함
- skip처리는 #으로 시작함
- 수집 실패 시 재시도 정책은 프로그램 명세서를 따름(최초 시도 실패 후 10초 간격 재시도 3회, 배치/주기 공통 적용)

## 스케쥴 예
- #[dlprem01-테스트개발][192.168.210.121][/tmp/check.out][배치|3|090001] -> skip
- [dlprem01-테스트개발][192.168.210.121][/tmp/check.out][배치|3|090001] -> 수요일 9시 00분 01초에 한번 수집
- [dlprem01-테스트개발][192.168.210.121][/tmp/check.out][배치|1-6|090001] -> 월~토요일 9시 00분 01초에 한번 수집
- [dlprem01-테스트개발][192.168.210.121][/tmp/check.out][배치|*|090001] -> 매일 9시 00분 01초에 한번 수집
- [dlprem01-테스트개발][192.168.210.121][/tmp/check.out][주기|*|090001|10|100001] -> 매일 9시 00분 01초부터 10분간격으로 10시 00분 01초까지 수집
- [dlprem01-테스트개발][192.168.210.121][/tmp/check.out][주기|1-5|090001|10|100001] -> 월~금요일 9시 00분 01초부터 10분간격으로 10시 00분 01초까지 수집

## 대상파일명 자리표시자 및 날짜 처리 정책

- 대상파일명에 아래 자리표시자를 포함하면 수집 시점에 실제 값으로 자동 치환됨. 자리표시자가 없는 고정 파일명(예: `.log`)은 치환 없이 그대로 사용함.
- 치환 우선순위: `yyyymmdd` → `mmdd` → `$` 순서 (`yyyymmdd`가 `mmdd`를 부분 문자열로 포함하므로 먼저 치환해야 결과가 깨지지 않음).

| 자리표시자 | 치환값 | 예시 |
|---|---|---|
| `yyyymmdd` | 수집일자 전체(yyyyMMdd) | `app.yyyymmdd.log` → `app.20260615.log` |
| `mmdd` | 수집일자의 월/일(MMdd) | `app.mmdd.log` → `app.0615.log` |
| `$` | 수집일자의 요일 숫자(0=일요일~6=토요일, 위 요일 표기 규칙과 동일 기준) | `sys0$.log` → (월요일 수집 시) `sys01.log` |
| (없음, 고정명) | 치환 없음 | `app.log` → `app.log` |

### 배치 처리 시

- 자리표시자 종류와 무관하게 스케줄 시작시간에 단 한 번, 그 시점에 치환된 경로의 파일 전체를 1번 라인부터 끝까지 읽음.
- `LAST_LINE_NUMBER`를 전혀 사용하지 않으므로 "날짜가 바뀌면 리셋"이라는 개념 자체가 없음 — 실행될 때마다 항상 전체를 새로 수집함.

### 주기 처리 시

- 기본 동작: 직전 성공 수집의 `LAST_LINE_NUMBER` 다음 줄부터 파일 끝까지 증분 수집. 단, 직전 성공 수집의 수집일자가 오늘과 다르면 `LAST_LINE_NUMBER`를 무시하고 1번 줄부터 다시 읽음(자정이 지나면 자동 리셋).
- `yyyymmdd` / `mmdd` / `$` 포함: 치환 결과가 매일(또는 매 요일) 다른 경로가 되므로, 날짜·요일이 바뀌면 물리적으로 다른 파일을 보게 됨 — 자정 리셋과 자연히 맞아떨어짐(어제 파일과는 별개 파일이라 처음부터 읽는 게 정상 동작).
  - 다만 `$`(요일)는 0~6 숫자가 7일마다 반복되므로, 같은 요일 파일이 매주 덮어쓰기(rotate)되는 운영 환경을 전제로 함. 덮어쓰기 없이 계속 누적되는 파일이면 지난주 끝 라인부터 잘못 이어 읽힐 수 있음.
- 고정명(`.log`, 자리표시자 없음): 경로가 매일 동일함. 자정이 지나도 같은 파일에 로그가 계속 누적되는 환경이면 경로 끝에 `+`를 붙여 날짜 리셋을 비활성화해야 함(예: `app.log+`). `+`가 없으면 파일이 안 바뀌어도 자정마다 1번 줄부터 재수집됨(=중복 수집처럼 보임).
- `+` 접미사는 `yyyymmdd` / `mmdd`와 함께도 사용 가능(예: `app.yyyymmdd.log+`) — 치환된 당일(또는 당월/일) 파일을 자정 넘어서까지 계속 이어쓰는 드문 경우에 사용.

  `precheck_collect\test_dataset`로 변경 — history seed `.sql`이 이미 쓰던 디렉토리와
  통일)
